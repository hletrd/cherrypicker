## Findings

### F1. CSV multi-line quoted fields not supported [MEDIUM]
**File**: `packages/parser/src/csv/shared.ts` (splitCSVLine)
**File**: `packages/parser/src/csv/index.ts` (content.split('\n'))
**Issue**: The parser splits content on `\n` first, then processes each line. RFC 4180 CSV allows fields containing newlines when enclosed in double quotes. Some Korean bank exports (especially those generated by Excel) may contain multi-line quoted fields (e.g., merchant names with line breaks, or memo fields). The current approach would split these across multiple "lines," corrupting the data.
**Impact**: Potential data corruption for CSV files with embedded newlines in quoted fields.

### F2. Positive sign (+) prefix in amounts not stripped [LOW]
**File**: `packages/parser/src/csv/shared.ts` (parseCSVAmount)
**File**: `packages/parser/src/xlsx/index.ts` (parseAmount)
**File**: `packages/parser/src/pdf/index.ts` (parseAmount)
**Issue**: Some Korean bank exports prefix positive amounts with a `+` sign (e.g., `+12,345`). The parsers only handle negative prefixes (`-`, `마이너스`, parentheses) but do not strip a leading `+` sign. `parseFloat("+12345")` correctly returns 12345, so this is only an issue if `+` is followed by a Won sign or other currency markers (e.g., `+₩12,345`) where the regex patterns would not match.
**Impact**: Low - `+` amounts parse correctly in most cases due to parseFloat behavior.

### F3. Delimiter detection counts characters inside quotes [LOW]
**File**: `packages/parser/src/detect.ts` (detectCSVDelimiter)
**Issue**: The delimiter detection counts ALL occurrences of each character, including those inside quoted fields. A CSV with many quoted commas (e.g., merchant names containing commas) could skew detection. However, since Korean bank CSVs rarely quote fields heavily, this is low priority.
**Impact**: Low - rarely causes incorrect detection in practice.

### F4. Adapter-factory skip condition differs from generic parser [LOW]
**File**: `packages/parser/src/csv/adapter-factory.ts` line 126
**Issue**: The adapter-factory skips rows with `if (!dateRaw && !merchantRaw) continue;` while the generic parser uses `if (!dateRaw && !merchantRaw && !amountRaw) continue;`. The adapter-factory would process a row with only an amount (no date, no merchant), producing a transaction with empty date/merchant. The generic parser correctly requires at least one of the three.
**Impact**: Low - produces a low-quality transaction with empty fields that would be filtered downstream.

### F5. PDF multi-line header support [DEFERRED]
**Issue**: PDF tables sometimes have headers split across two rows (e.g., one row has "이용", next row has "일"). This requires row-merging heuristics that would add significant complexity.
**Impact**: Low - rare in practice, most PDFs have single-line headers.

### F6. No test for CSV with tab/semicolon/pipe delimiters in CSV-specific tests [LOW]
**File**: `packages/parser/__tests__/csv.test.ts`
**Issue**: While delimiter detection is tested in detect.test.ts, there are no CSV parsing tests that exercise the tab, semicolon, or pipe delimiter paths through parseCSV.

### F7. No test for CSV with Windows line endings (CRLF) [LOW]
**File**: `packages/parser/__tests__/csv.test.ts`
**Issue**: Korean bank exports from Windows may use CRLF line endings. The parser splits on `\n` which would leave trailing `\r` characters in field values. The `splitCSVLine` trims fields, which would strip `\r`, but the header detection regex patterns are not tested against `\r`-suffixed values.

### F8. SUMMARY_ROW_PATTERN missing "소계" English equivalent "subtotal" [LOW]
**File**: `packages/parser/src/csv/column-matcher.ts`
**Issue**: The pattern matches "소계" (Korean subtotal) but does not have the English equivalent "subtotal". Adding English keywords improves parity for international exports.
